## 一、项目概述

### 1.1 一句话定位

**一款2D俯视角、纯PVE、梗文化包装的轻量搜打撤微信小游戏。**

对标《修勾逃生巨人危机》的轻度IAA路线，而非《无限轮回》的重度IAP路线。核心卖点：碎片化搜打撤体验 + 沙雕/梗文化传播性 + 微信社交裂变。

### 1.2 目标用户

| 维度 | 画像 |
|---|---|
| 年龄 | 16-35岁，Z世代与年轻职场人 |
| 场景 | 通勤、午休、睡前等碎片化时间 |
| 偏好 | 喜欢搜打撤/Roguelike但没时间玩重度端游；喜欢沙雕/梗文化 |
| 付费意愿 | 低——以看广告为主，少量轻度内购 |

## 三、核心玩法设计

### 3.1 搜打撤"不可能三角"的取舍

搜打撤三要素存在本质矛盾（参考《搜打撤的不可能三角》理论）：提升任一要素的体验必然削弱其他两个。我们的取舍：

| 要素 | 权重 | 设计倾向 |
|---|---|---|
| **搜** | ★★★★☆ | 主要驱动力——搜集的"开盲盒"爽感是碎片化用户的核心诉求 |
| **打** | ★★★☆☆ | 辅助调味——战斗提供紧张感但不要求高操作，自动射击降低门槛 |
| **撤** | ★★★★★ | 情绪高潮——撤离是整局的情绪顶点，必须做出心跳感 |

### 3.2 单局完整流程

```
┌─────────────────────────────────────────────────┐
│                  单局流程（2-3分钟）               │
├─────────────────────────────────────────────────┤
│                                                   │
│  1. 准备阶段（5秒）                                │
│     → 选择携带武器（从已解锁武器中选1把）            │
│     → 显示地图预览（本局地图的大致结构）              │
│     → 倒计时3秒，空投落地                           │
│                                                   │
│  2. 搜索阶段（60-90秒）                            │
│     → 操控角色在地图中移动                          │
│     → 靠近搜索点自动触发搜索（进度条0.5-1.5秒）      │
│     → 获得物资：金币/材料/武器碎片/Buff道具          │
│     → 随机遭遇巡逻怪物，自动射击消灭                │
│     → 击杀怪物掉落额外物资                          │
│     → 危险度随时间递增（每30秒提升一级）              │
│                                                   │
│  3. 撤离阶段（30秒）                               │
│     → 地图上随机刷出1-2个撤离点（发光标记）           │
│     → 玩家需移动到撤离点                            │
│     → 抵达后触发15秒倒计时                          │
│     → 倒计时期间怪物加速涌来（最后冲刺感）            │
│     → 存活到倒计时结束 → 撤离成功                    │
│     → 中途死亡 → 失去本局全部搜集物资                │
│                                                   │
│  4. 结算阶段（5秒）                                │
│     → 成功：物资转化为永久货币/材料                  │
│     → 失败：提供"看广告复活"选项                    │
│     → 显示本局数据（搜集量/击杀数/好友排名）          │
│     → "再来一局" / "返回主界面"                     │
│                                                   │
└─────────────────────────────────────────────────┘
```

### 3.3 危险度递增机制

危险度是控制单局节奏的核心变量，替代传统搜打撤的"毒圈"概念：

| 时间 | 危险等级 | 表现 |
|---|---|---|
| 0-30秒 | Lv.1 平静 | 怪物少、移动慢、伤害低；搜索点丰富 |
| 30-60秒 | Lv.2 警戒 | 怪物数量+50%，出现精英怪；屏幕边缘开始泛红 |
| 60-90秒 | Lv.3 危险 | 怪物数量翻倍，移速加快；撤离点刷新提示出现 |
| 90秒+ | Lv.4 崩溃 | 怪物无限涌出，伤害极高；不撤离必死 |

设计意图：
- 前30秒给玩家"安全搜刮"的舒适期，建立正反馈
- 中间60秒制造"要不要再多搜一点"的贪婪博弈
- 90秒后强制逼迫撤离，制造高潮紧张感

### 3.4 怪物刷新规则（DangerManager行为） 【M1必须】

```
DangerManager 每个逻辑帧做两件事：
  1. 更新危险等级（根据已过时间）
  2. 按规则刷新怪物

刷新参数（按危险等级）：
  | 危险等级 | 场上最大存活数 | 刷新间隔 | 每次刷新数量 | 怪物类型 |
  |----------|---------------|---------|-------------|---------|
  | Lv.1     | 5             | 8秒     | 1           | 小兵     |
  | Lv.2     | 8             | 5秒     | 1-2         | 小兵+巡逻怪 |
  | Lv.3     | 12            | 3秒     | 2           | 小兵+巡逻怪+精英怪 |
  | Lv.4     | 20            | 1.5秒   | 2-3         | 全类型   |

刷新位置选择：
  1. 从 enemySpawnPoints 中筛选距玩家 > 8格的点（不在玩家视野内刷怪）
  2. 从筛选结果中随机选一个
  3. 如果所有点都距玩家 < 8格（小地图/玩家在中间） → 选最远的那个

刷新逻辑（伪代码）：
  spawnTimer += dt
  if (aliveEnemyCount < maxAlive && spawnTimer >= spawnInterval) {
    spawnTimer = 0
    const count = randomInRange(spawnCountMin, spawnCountMax)
    for (i = 0; i < count; i++) {
      const pos = pickSpawnPoint(player.pos, enemySpawnPoints)
      const type = pickEnemyType(currentDangerLevel)
      EnemyManager.spawn(type, pos)
    }
  }

怪物被击杀后：
  aliveEnemyCount-- （自然减少，到下次刷新间隔时补充）
  不做立即补充（给玩家清怪后的短暂安全窗口）
```

### 3.5 撤离交互规则 【M1必须】

```
撤离区域：
  大小: 3×3格（以撤离点标记为中心），地板上有明显光圈标识
  出现时机: 危险度达到Lv.3（60秒）时刷出，位置见5.3节地图生成规则
  出现动画: 光柱从天而降 + 屏幕提示"撤离点已出现" + 小地图标记

撤离流程：
  1. 玩家进入3×3撤离区域 → HUD出现"撤离"按钮
  2. 点击按钮 → 开始15秒倒计时
  3. 倒计时期间：
     - 玩家可以自由移动和射击（不要求站在区域内）
     - 倒计时一旦启动就不会暂停或取消
     - 怪物刷新加速（间隔减半），营造最后冲刺的紧张感
  4. 倒计时归零 → 立即判定：
     - 玩家在撤离区域内（3×3范围） → 撤离成功 → 进入结算
     - 玩家不在撤离区域内 → 倒计时结束后撤离点消失
       → 30秒后刷出新的撤离点（给第二次机会）
       → 第二次失败 → 不再刷新（Lv.4会淹没玩家）
  5. 玩家死亡 → 直接进入失败结算（无论倒计时状态）

设计考量：
  - 倒计时启动后不取消 → 简化实现，避免"反复进出刷倒计时"的exploit
  - 可以离开区域 → 允许玩家在倒计时期间走位打怪，增加操作空间
  - 最终判定需在区域内 → 保留"最后冲回来"的紧张感
```

### 3.6 "贪婪博弈"——核心心理钩子

搜打撤的灵魂在于"我要不要再多搜一个箱子"的决策：

- 每多搜一个箱子 → 收益增加（可能搜到稀有物品）
- 每多停留一秒 → 风险增加（危险度上升，怪物变强）
- 死亡代价 → 本局全部物资清零

通过以下设计强化这个博弈：
- **高价值搜索点放在地图深处**（远离撤离点）
- **稀有物品有独特音效和光效**（勾引玩家冒险）
- **背包容量有限**（不能无限搜，必须取舍）
- **死亡时显示"你带着XX物品被击杀"**（强化损失感）

## 四、操控方案

### 4.1 操控布局（竖屏）

```
┌─────────────────────────┐
│  [HP条]  [危险度]  [背包] │  ← HUD顶栏
│                           │
│                           │
│         游戏画面           │  ← 相机跟随玩家
│        （俯视角）          │
│                           │
│                           │
│   ┌───┐           [技能]  │  ← 左摇杆 + 右侧技能按钮
│   │ ◎ │                   │
│   └───┘           [撤离]  │  ← 撤离点附近出现撤离按钮
└─────────────────────────┘
```

### 4.2 操控方案详细

| 操作 | 方式 | 说明 |
|---|---|---|
| **移动** | 左下虚拟摇杆（连续方向） | 拇指拖拽，松手停下 |
| **射击** | 自动射击 | 进入武器射程内自动攻击最近敌人，玩家专注移动和走位 |
| **技能** | 右下技能按钮（1个主动技能） | 冷却后点击释放，如手雷/闪光弹/冲刺 |
| **搜索** | 自动触发 | 靠近搜索点自动开始搜索（进度条），无需点击 |
| **拾取** | 自动拾取 | 走过掉落物自动拾取进背包 |
| **撤离** | 撤离区域内出现按钮 | 进入撤离区域后点击"撤离"按钮启动倒计时 |

设计原则：
- **左手管移动，右手管技能**——与主流MOBA操控一致，零学习成本
- **自动射击+自动拾取**——剥离重复操作，让玩家专注"走位"和"决策"
- **单局只有1个主动技能**——最小化操作复杂度

### 4.3 虚拟摇杆实现规格 【M1必须】

```
摇杆参数：
  底盘半径: 60px（设计坐标）
  摇杆圆心半径: 20px
  死区: 底盘半径的15%（< 9px的拖拽不触发移动，防误触）
  最大拖拽距离: 60px（超出部分摇杆跟随手指但输出值钳制为1.0）
  位置: 左下角，距屏幕底部100px，距左边80px
  触发方式: touchstart在底盘区域内 → 激活 → touchmove更新方向 → touchend释放

输出值：
  direction: Vec2（归一化方向向量，长度0-1）
  magnitude: number（0-1，0=死区内/未触发，1=满幅拖拽）

多点触控处理：
  摇杆绑定第一个落在左半屏的touchId
  技能按钮绑定第一个落在右半屏的touchId
  两者独立跟踪，互不干扰
```

### 4.4 自动射击目标选择算法 【M1必须】

```
每帧执行（逻辑帧30fps）：

1. 收集射程内所有敌人（以玩家为圆心，半径 = 武器射程）
2. 过滤掉被墙壁遮挡的敌人（射线检测：从玩家到敌人的直线路径上是否有碰撞格）
3. 从可见敌人中选择目标：
   - 优先级1: 当前锁定目标（如果仍在射程内且可见）→ 保持锁定（防抖动）
   - 优先级2: 距离最近的敌人
   - 优先级3: 无目标 → 停火
4. 锁定切换冷却: 200ms（切换目标后200ms内不再切换，避免频繁跳目标）

射线遮挡检测（简化版）：
  从玩家格子到目标格子，用Bresenham直线算法遍历经过的格子
  任一格子为墙壁(collisionGrid=true) → 判定为遮挡
```

### 4.5 操控手感参数 【M1必须】

| 参数 | 值 | 说明 |
|---|---|---|
| 移动速度 | 5格/秒（160px/秒） | 背包满时降至3.5格/秒 |
| 自动射击范围 | 武器射程（手枪4格/霰弹2格/冲锋枪4格） | 以玩家为圆心的圆形区域 |

## 4.6 碰撞与移动模型 【M1必须】

```
实体尺寸（圆形碰撞体，所有实体均用圆简化，避免AABB旋转问题）：
  玩家:   半径 10px（0.3125格），中心 = 精灵中心
  小兵:   半径 10px
  巡逻怪: 半径 10px
  胖子:   半径 14px（体型大）
  精英怪: 半径 12px
  子弹:   半径 3px
  掉落物: 半径 6px（仅用于拾取检测，不参与物理碰撞）

亚格移动+网格碰撞检测：
  玩家坐标是像素级浮点数（不是格子级整数）
  每帧移动: newPos = pos + direction * speed * dt

  碰撞检测步骤（每次移动前）：
  1. 计算移动后玩家圆形碰撞体覆盖的所有网格（通常2×2或3×3个格子）
  2. 检查这些格子中是否有 collisionGrid=true 的墙壁
  3. 如果有碰撞 → 执行分轴处理（wall sliding）

Wall Sliding（关键体验）：
  不直接拒绝移动，而是分X/Y轴分别尝试：
  
  // 先尝试完整移动
  if (!collides(pos.x + dx, pos.y + dy)) {
    pos.x += dx; pos.y += dy;  // 无碰撞，正常移动
  }
  // 完整移动碰撞 → 分轴尝试
  else if (!collides(pos.x + dx, pos.y)) {
    pos.x += dx;  // 只走X轴（沿水平墙壁滑动）
  }
  else if (!collides(pos.x, pos.y + dy)) {
    pos.y += dy;  // 只走Y轴（沿垂直墙壁滑动）
  }
  // 两轴都碰 → 完全不动

  collides(x, y) 函数：
    检查以(x,y)为圆心、玩家半径覆盖的所有网格
    任一格子为墙 → 返回true

  效果：玩家斜向贴墙时会自动沿墙壁方向滑动，手感流畅
```

### 5.2 Tile类型

| Tile类型 | 标识 | 说明 |
|---|---|---|
| 地板 | `0` | 可通行区域 |
| 墙壁 | `1` | 不可通行，阻挡视线和子弹 |

| 陷阱 | `7` | 踩上触发伤害（可选） |

### 5.3 程序化地图生成 【M1必须】

采用 **BSP（Binary Space Partition）+ 走廊连接** 算法，这是Roguelike领域最成熟、最不容易生成不可达地图的方案：

```
算法步骤（伪代码）：

Step 1: BSP分割
  root = 整个40×40区域
  递归分割（随机水平或垂直切一刀），直到叶子节点 >= 8×8 且 <= 16×16
  分割时确保每个子区域最小边 >= 8（容纳最小房间4×4 + 边距）
  最终得到5-8个叶子节点

Step 2: 在每个叶子节点内生成房间
  房间大小 = 叶子区域 - 随机边距(1-2格)
  最小房间: 4×4, 最大房间: 8×8
  房间内全部填充为地板(0)

Step 3: 连接房间（自底向上）
  从BSP树叶子向上回溯：
  每个非叶子节点的两个子节点，取各自区域内房间的中心点
  用L形走廊连接两个中心点（先水平再垂直，或反之随机）
  走廊宽度: 2格（保证通行舒适度）
  → BSP结构保证所有房间必定连通，无需额外检验

Step 4: 放置功能点
  出生点: 离地图几何中心最近的房间中央
  撤离点: 离出生点曼哈顿距离最远的1-2个房间（距离>20格）
  搜索点: 每个房间内随机2-4个，贴墙放置（模拟柜子/箱子）
  高价值搜索点: 离撤离点最远的2个房间各放1个
  怪物刷新点: 走廊中段 + 离出生点>15格的房间入口
  钥匙门: 可选，放在通往高价值房间的走廊上

Step 5: 难度参数化（不同地图模板）
  废弃仓库: 房间大(6-8), 走廊短, 怪物刷新点少
  闹鬼学校: 房间中(4-6), 走廊长且多拐角, 怪物刷新点中
  地下实验室: 房间小(4-5), 走廊窄, 怪物刷新点密集
```

**为什么选BSP而不是随机撒房间**：
- 随机撒房间需要处理重叠检测、可达性验证——边界情况多，容易出bug
- BSP的树结构天然保证无重叠、全连通，代码更简单可靠
- BSP在Roguelike领域有大量参考实现（NetHack、Brogue等），算法成熟

**数据结构**：

```typescript
interface MapData {

  height: number;               // 40
  tiles: number[][];            // 二维数组，值为Tile类型枚举
  collisionGrid: boolean[][];   // true=不可通行（墙/门），碰撞检测用
```

### 6.3 怪物AI状态机 【M1必须】

所有怪物共用一个状态机框架，不同怪物类型覆写参数和行为：

```
状态枚举: IDLE → PATROL → CHASE → ATTACK → DEAD

通用参数：
  感知半径: 发现玩家的最大距离（格数）
  感知方式: 圆形范围（不做视锥，简化实现）
  追踪上限: 追多远放弃（格数），超出则回到PATROL/IDLE
  攻击距离: 进入此距离切换到ATTACK
  攻击间隔: 两次攻击之间的冷却时间

┌──────┐  感知到玩家  ┌───────┐  进入攻击距离  ┌────────┐
│ IDLE │ ──────────→ │ CHASE │ ───────────→ │ ATTACK │
└──┬───┘             └───┬───┘              └────┬───┘
   │有巡逻路径           │超出追踪上限            │目标离开攻击距离
   ▼                     ▼                      ▼
┌────────┐          回到PATROL/IDLE          回到CHASE
│ PATROL │
└────────┘

各怪物类型参数：
  小兵:   感知5格, 追踪10格, 攻击距离1格(近战), 攻击间隔1s, 无寻路(直线移动)
  巡逻怪: 感知6格, 追踪12格, 攻击距离3格(射击), 攻击间隔1.5s, A*寻路
  胖子:   感知4格, 追踪无限, 攻击距离1格(自爆), 直线冲锋, 3秒后自爆
  精英怪: 感知8格, 追踪15格, 攻击距离4格(射击), 攻击间隔2s, A*寻路
  Boss:   感知=房间范围, 不离开房间, 多阶段攻击(M2再细化)
```

**A*寻路性能控制** 【M1必须】：
- 每只怪物每500ms重新计算一次路径（不是每帧）
- A*搜索节点上限200，超出则降级为"向玩家方向直线移动，遇墙停"
- 同时寻路的怪物上限5只，超出的排队等下一个500ms周期
- 路径缓存：目标位置变化<2格时复用旧路径

### 6.4 伤害与战斗规则 【M1必须】

```
伤害公式：
  最终伤害 = 武器基础伤害 × (1 + 等级加成%) × (1 + Buff加成%) 
  （怪物无防御值，简化计算）

受击规则：
  无敌帧: 受伤后500ms内不再受伤
  击退: 受伤瞬间向攻击反方向推移0.5格（不穿墙，碰墙则停）

死亡规则：
  HP≤0 → 进入死亡状态
  播放死亡动画（倒地0.5秒）
  弹出结算界面（或广告复活选项）
  死亡期间游戏逻辑暂停（怪物冻结）

暂停与后台：
  收到微信onHide → 游戏逻辑暂停（ticker.stop）
  收到微信onShow → 弹出"继续游戏"确认 → ticker.start
  暂停期间存储当前RunState（如果需要断点续玩——MVP可不做，直接结算当局失败）
```

## 七、Roguelike构筑系统

### 7.1 局内Buff——搜索过程中的随机强化

搜索容器时有概率获得临时Buff（仅本局有效），这是Roguelike的核心体验：

| Buff类别 | 示例 | 触发概率 |
|---|---|---|
| 攻击强化 | 攻击力+20%、射速+15%、暴击率+10% | 30% |
| 防御强化 | 最大生命+20%、受伤减免15%、自动回血 | 20% |
| 移速强化 | 移速+20%、冲刺距离+50% | 15% |
| 特殊效果 | 子弹穿透、击杀回血、范围溅射 | 10% |
| 搜索强化 | 搜索速度+50%、搜索额外掉落 | 15% |
| 稀有（金色） | 双枪模式、无限子弹(10秒)、分身 | 5% |

### 7.2 Build流派引导

虽然是轻度游戏，但通过Buff组合让玩家自然形成"流派"偏好：

| 流派 | 核心Buff | 玩法风格 |
|---|---|---|
| 火力型 | 攻击力+暴击+穿透 | 快速清怪，适合高难地图 |
| 坦克型 | 生命+减伤+回血 | 硬抗，适合撤离阶段的最后冲刺 |
| 跑酷型 | 移速+冲刺+搜索速度 | 快速搜完就跑，适合速通 |

### 7.3 Buff叠加与管理规则 【M1必须】

```
叠加规则：
  同类型数值Buff: 可叠加，效果相加（攻击+20% + 攻击+20% = 攻击+40%）
  同类型上限: 同一属性最多叠3层（防止数值崩溃）
  特殊效果Buff: 不叠加（子弹穿透获得2次 = 还是穿透1次，不叠加）
  互斥Buff: 无（简化设计，不做互斥）

持有上限: 无上限（但单局搜索点有限，实际不会超过6-8个Buff）

获取方式:
  搜索容器时 → 10-30%概率掉落一个随机Buff → 自动生效
  不做"选择"界面（简化交互，碎片化用户不想做太多决策）

数据结构:
  interface BuffInstance {
    id: string;           // 'atk_up_20' 等
    type: 'stat' | 'special';
    stat?: string;        // 'atk' | 'def' | 'spd' | 'atkSpeed' | 'searchSpeed'
    value?: number;       // +0.2 表示+20%
    special?: string;     // 'pierce' | 'lifesteal' | 'splash'
    stackCount: number;   // 当前叠加层数
  }
```

```
  Buff:  10%概率

  金币: 30 × 10 = 300金币
  碎片: 1 × 10 = 10碎片
  强化石: 0.3 × 10 = 3个

  金币: +50-100
  碎片: +2-5
```

### 13.2 游戏主循环 【M1必须】

```
BattleScene.onEnter() 时启动，通过 Game.ticker 驱动：

Game.ticker.add((delta) => {
  const dt = delta / 60;  // 归一化为秒（PixiJS ticker delta基于60fps）

  if (gameState === 'paused' || gameState === 'dead') return;

  // 1. 输入处理
  InputSystem.update();

  // 2. 游戏逻辑（固定30fps更新）
  logicAccumulator += dt;
  while (logicAccumulator >= LOGIC_STEP) {  // LOGIC_STEP = 1/30
    RunManager.update(LOGIC_STEP);          // 单局状态机
    DangerManager.update(LOGIC_STEP);       // 危险度递增
    PlayerManager.update(LOGIC_STEP);       // 玩家移动+自动射击
    EnemyManager.update(LOGIC_STEP);        // 怪物AI+寻路
    BulletManager.update(LOGIC_STEP);       // 子弹飞行+碰撞
    CombatManager.update(LOGIC_STEP);       // 伤害结算
    LootManager.update(LOGIC_STEP);         // 掉落物检测拾取
    ExtractionManager.update(LOGIC_STEP);   // 撤离点检测
    logicAccumulator -= LOGIC_STEP;
  }

  // 3. 渲染更新（跟随设备帧率，可以是60fps）
  CameraSystem.update(dt);                  // 相机跟随（平滑插值）
  FogOfWarSystem.update();                  // 战雾重绘
  ParticleSystem.update(dt);                // 粒子效果
  TweenManager.update(dt);                  // UI动画
});
```

逻辑帧与渲染帧分离的原因：
- 逻辑帧固定30fps → 碰撞/AI行为在不同设备上表现一致
- 渲染帧跟随设备 → 高端机60fps流畅，低端机30fps也不影响游戏逻辑
- 相机用插值平滑 → 即使逻辑帧30fps，画面移动也流畅无顿挫

### 13.3 相机系统 【M1必须】

```
实现方式：
  地图是一个大Container（mapContainer），所有Tile/怪物/掉落物都在其中
  相机效果 = 移动mapContainer的pivot，使玩家始终在屏幕中央

CameraSystem.update(dt):
  targetX = player.x
  targetY = player.y
  
  // 平滑跟随（lerp插值，避免顿挫）
  camera.x += (targetX - camera.x) * 0.1
  camera.y += (targetY - camera.y) * 0.1
  
  // 边界约束（不能看到地图外的黑边）
  camera.x = clamp(camera.x, viewportW/2, mapW - viewportW/2)
  camera.y = clamp(camera.y, viewportH/2, mapH - viewportH/2)
  
  mapContainer.pivot.set(camera.x, camera.y)
  mapContainer.position.set(screenW/2, screenH/2)  // 设计坐标

HUD层独立：
  HUD（血条/摇杆/按钮）放在独立的Container，不跟随相机
  层级: stage → mapContainer(随相机) → hudContainer(固定) → overlayContainer(弹窗)
```

### 13.4 战雾方案 【M1必须，验证性能】

```
方案：PixiJS Graphics + 圆形遮罩

实现：
  1. fogGraphics = new PIXI.Graphics()，放在mapContainer最上层
  2. 每帧重绘：
     fogGraphics.clear()
     fogGraphics.beginFill(0x000000, 0.85)  // 半透明黑色
     fogGraphics.drawRect(0, 0, mapWidth, mapHeight)  // 覆盖全图
     fogGraphics.beginHole()
     fogGraphics.drawCircle(player.x, player.y, viewRadius)  // 扣出可见区域
     fogGraphics.endHole()
     fogGraphics.endFill()
  
  viewRadius: 4格 = 128px（设计坐标）

性能备选方案（如果Graphics每帧重绘卡顿）：
  改用AlphaMaskFilter + 一张预渲染的渐变圆形Texture
  圆形Texture固定大小，跟随玩家位置移动
  → 零重绘开销，只需更新mask的position
```

### 13.5 关键技术方案

| 技术点 | 方案 | 复杂度 |
|---|---|---|
| **地图渲染** | 将40×40 Tile预渲染为一张大的Container，相机通过偏移Container.pivot实现滚动 | 低 |
| **碰撞检测** | 维护一个 `boolean[][]` 碰撞网格，移动前查表判断，O(1)效率 | 低 |
| **A*寻路** | 自实现网格A*，限制搜索节点数上限（200），避免大地图卡顿 | 中 |
| **对象池** | 子弹、伤害数字、粒子使用对象池复用，避免GC卡顿 | 低 |
| **战雾** | PixiJS Graphics绘制全屏黑色遮罩，以玩家位置为圆心扣出可见区域 | 低 |
| **帧率控制** | 游戏逻辑固定30fps更新（ticker deltaTime归一化），渲染跟随设备帧率 | 低 |
| **包体控制** | 目标主包 < 4MB，图片走分包/CDN，代码bundle约300-500KB | 低 |

| 指标 | 目标 | 策略 |

| 帧率 | 稳定30fps（低端机）/ 60fps（中高端） | 对象池、批量渲染、逻辑帧与渲染帧分离 |
| 内存 | < 150MB | 纹理按需加载、离屏对象销毁 |
| 主包大小 | < 4MB | 代码tree-shaking + 图片分包 |
| 启动时间 | < 3秒 | 核心资源内联、非核心资源异步 |
